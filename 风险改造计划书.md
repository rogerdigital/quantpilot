# QuantPilot 风险改造计划书

## 结论

当前 QuantPilot 已经具备完整的平台骨架和较宽的功能覆盖面，但工程成熟度还处于快速平台化阶段。主要风险集中在可验证性、类型约束、API 路由合同、Worker 调度可靠性，以及真实交易与模拟模式边界不够清晰。

改造顺序应严格按以下 5 个阶段推进：

1. 恢复 `main` 分支本地可验证性。
2. 治理核心路径的 `@ts-nocheck`。
3. 收紧后端路由与 Context 合同。
4. 增强 Worker 调度可靠性。
5. 明确真实交易与 simulated fallback 边界。

建议每个阶段独立提交或独立 PR。不要把 5 条风险混成一个大提交，否则 review、回滚和定位问题都会变困难。

## 阶段 4：增强 Worker 调度可靠性

### 目标

一个 Worker 任务失败或超时，不应阻塞整轮 tick。每个任务的成功、失败、超时都要有稳定结果，便于日志、测试和后续监控。

### 修改范围

- `apps/worker/src/config.ts`
- `apps/worker/src/runtime/worker-runtime.ts`
- `apps/worker/test/worker-tasks.test.ts`
- `apps/worker/test/worker-workflow-e2e.test.ts`

### 实施步骤

1. 在 Worker config 增加两个配置项：

```ts
export type WorkerConfig = {
  name: string;
  intervalMs: number;
  once: boolean;
  notificationBatchSize: number;
  riskScanBatchSize: number;
  workflowBatchSize: number;
  taskTimeoutMs: number;
  continueOnTaskFailure: boolean;
};
```

2. 读取环境变量：

```ts
taskTimeoutMs: readNumber(process.env.QUANTPILOT_WORKER_TASK_TIMEOUT_MS, 10000),
continueOnTaskFailure: process.env.QUANTPILOT_WORKER_CONTINUE_ON_FAILURE !== '0',
```

3. 新增任务包装器：

```ts
async function runTaskWithTimeout(
  taskName: string,
  task: () => Promise<WorkerTaskResult>,
  config: WorkerConfig
): Promise<WorkerTaskResult> {
  let timeoutId: NodeJS.Timeout | null = null;

  const timeout = new Promise<WorkerTaskResult>((resolve) => {
    timeoutId = setTimeout(() => {
      resolve({
        worker: config.name,
        kind: taskName,
        timestamp: new Date().toISOString(),
        summary: `Task timed out after ${config.taskTimeoutMs}ms`,
        ok: false,
      });
    }, config.taskTimeoutMs);
  });

  try {
    const result = await Promise.race([task(), timeout]);
    return {
      ...result,
      ok: result.ok !== false,
    };
  } catch (error) {
    return {
      worker: config.name,
      kind: taskName,
      timestamp: new Date().toISOString(),
      summary: error instanceof Error ? error.message : 'Unknown worker task failure',
      ok: false,
    };
  } finally {
    if (timeoutId) clearTimeout(timeoutId);
  }
}
```

4. 第一版保持任务顺序不变，只做失败隔离：

```ts
const taskEntries = [
  ['workflow-maintenance', () => runWorkflowMaintenanceTask(config)],
  ['workflow-execution', () => runWorkflowExecutionTask(config)],
  ['scheduler-tick', () => runSchedulerTickTask(config)],
  ['risk-scan', () => runRiskScanTask(config)],
  ['heartbeat', () => runHeartbeatTask(config)],
  ['monitoring-scan', () => runMonitoringScanTask(config)],
  ['notification-dispatch', () => runNotificationDispatchTask(config)],
];
```

5. 后续如果要并行，只允许并行无状态或低冲突任务：

- `heartbeat`
- `monitoring-scan`
- `notification-dispatch`

不要把 `workflow-execution` 和 `scheduler-tick` 并行，避免控制面状态竞争。

6. 增加测试：

- 单任务 throw 时后续任务仍执行
- 单任务 timeout 时 tick 返回失败结果
- `once` 模式仍执行一轮后退出
- `QUANTPILOT_WORKER_CONTINUE_ON_FAILURE=0` 时失败后停止后续任务

## 阶段 5：明确真实交易与 simulated fallback 边界

### 目标

让系统和用户都能明确识别当前处于哪种交易模式，避免把 simulated fallback 误判为 paper/live 交易链路可用。

### 修改范围

- `.env.example`
- `scripts/check-runtime-env.ts`
- `apps/web/src/app/config/runtime.ts`
- `apps/web/src/app/providers/broker.ts`
- `apps/web/src/app/providers/marketData.ts`
- `apps/api/src/gateways/alpaca.ts`
- `README.zh-CN.md`
- `README.md`

### 实施步骤

1. 定义交易运行模式：

```ts
export type TradingRuntimeMode = 'simulated' | 'paper' | 'live';
```

2. `.env.example` 增加：

```env
QUANTPILOT_TRADING_MODE=simulated
ALPACA_USE_PAPER=true
QUANTPILOT_LIVE_TRADING_ACK=
```

3. `scripts/check-runtime-env.ts` 增加校验规则：

- `simulated`：允许 Alpaca key 为空
- `paper`：必须存在 `ALPACA_KEY_ID` 和 `ALPACA_SECRET_KEY`，且 `ALPACA_USE_PAPER=true`
- `live`：必须存在 Alpaca key，且必须满足：

```env
QUANTPILOT_LIVE_TRADING_ACK=I_UNDERSTAND_LIVE_TRADING_RISK
ALPACA_USE_PAPER=false
```

4. API health 返回运行模式：

```json
{
  "tradingMode": "simulated",
  "brokerConfigured": false,
  "liveTradingEnabled": false
}
```

5. 前端 provider 状态文案区分：

- `Simulated`
- `Paper via Alpaca`
- `Live via Alpaca`

6. 前端如果进入 `live` 模式但 health 返回 `liveTradingEnabled=false`，必须显示阻断状态，不允许隐藏 fallback。

7. 文档更新：

- README 保留“不是生产级实盘交易系统”的声明
- 增加 simulated、paper、live 三种模式说明
- 增加进入 paper/live 前的配置清单
- 明确浏览器端不保存真实 broker 密钥

### 验证命令

```bash
npm run check:runtime-env
npm run test:api
npm run test:web
npm run build
```

建议额外手动验证：

```bash

QUANTPILOT_TRADING_MODE=paper npm run check:runtime-env -- --env-file .env.example

```

预期：

- `simulated` 通过
- `paper` 因缺少 Alpaca key 失败
- `live` 因缺少 Alpaca key 和 live ack 失败

## 最终成功标准

完成 5 个阶段后，应满足：

- `npm run verify` 通过。
- Web 能解析 `@quantpilot/ui`。
- 核心 API route dispatcher 不再使用 `@ts-nocheck`。
- Worker 单任务失败或超时不会拖垮整轮 tick。
- 用户能在 UI、API health、配置校验和 README 中明确看到当前交易模式。
- paper/live 模式缺少必要配置时会明确失败，而不是静默 fallback 到 simulated。
